# Semantic ARIS / BPM - Тезисы
Semantic BPM / EA на примере semantic VAD [ver9d](https://github.com/bpmbpm/rdf-grapher/tree/main/ver9d)  
Подробно см. https://github.com/bpmbpm/rdf-grapher/blob/main/introduction/introduction.md
## A. Концепт

### Для пользователя
- **Внешне** — классическая ARIS-подобная система (аналог ARIS Toolset, Business Studio, Sparx Architect и т.п.)
- **Базовый функционал** — типовые компоненты:
  - Моделер (создание и редактирование схем)
  - Репозитарий (хранение объектов моделирования)
  - Паблишер (визуализация и публикация)
  - Отчеты (запросы к репозитарию)

### "Под капотом" — стек Linked Data 
- **Технологическая основа:** Semantic Web, Wikidata, semantic MediaWiki и подобные технологии \ системы
- **Хранилище:** Quadstore (triplestore) для RDF данных (in-memory \ graph database)
- **SPARQL** - как основной язык, т.е. все запросы пользователя в меню системы преобразуют в SPARQL, т.е. все функции (доступные через из UI, отчеты и т.п.) классической ARIS-подобной системы реализуются как  SPARQL к quadstore (репозитарий ARIS-подобной системы). Кроме того, сама semantic BPM \ ARIS система реализована преимущественно на SPARQL (SPARQL - ориентированный подход программирования). 

### Внешние преимущества

#### 1. Абсолютный контроль над объектами моделирования / репозитарием объектов / связями объектов
- Любой объект (процесс, роль, документ и т.п.), его атрибуты, группировки — всё доступно через SPARQL-запрос
- Не нужно программировать логику анализа — она декларативно описывается в запросах. : Любой объект, его атрибут, группировка – выполняются на основе SPARQL-запроса к quadstore (triplestore, хранилище).  
- **Пример:** Построение матрицы RACI по SPARQL-запросу к схемам (к репозитарию): кто и как в каких процессах участвует (вкл. владение процессом).
- **Для пользователей без знания SPARQL:** Конструктор запросов (основной режим работы semantic ARIS)

#### 2. Механизм рассуждений (Semantic Reasoning)
- Запросы с учётом правил вывода через semantic reasoner, т.е. семантическим reasoner обрабатывает сязапросы с учетом правил вывода. 
- **Пример:** Утверждения: "Петя — сын Васи", "Вася — сын Вовы". Reasoner автоматически выводит: "Петя — внук Вовы"

#### 3. Готовый арсенал LD-инструментов
- Использование существующих Linked Data (RDF, RDFS, OWL, SPARQL, SHACL и др.) инструментов для формирования хранилища, анализа и редактирования атрибутов объектов
- Совместимость с экосистемой Semantic Web
- COTS (LD-lib) vs proprietary development (ARIS, Bussiness Studio etc)

### Технологические преимущества (внутренние)

#### 1. Унификация и совместимость
- LD-системы моделирования обеспечивают высокий уровень совместимости. Этого нет ни в BPMS типа ARIS, ни в BPMN-engine. Стандартов обеспечения совместимости в мире класического BPMS (ARIS) и исполняеиого BPMN (camunda и прочее) сегодня нет.
- Использование загружаемых онтологий для стандартизации. Не нужны реляцонные базы (у кажого свой "велосипед") и SQL и т.п. нужно только описание в виде онтологии и SPARQL. 

#### 2. Быстрота разработки
- Большой выбор библиотек (включая JavaScript):
  - N3.js — парсинг RDF
  - Comunica — SPARQL движок
  - In-memory quadstore
  - Semantic reasoner
- Готовые решения для triplestore, SPARQL, reasoning

#### 3. Прозрачность логики
- Semantic ARIS разработан на SPARQL (UI на JavaScript)
- Под каждой кнопкой — соответствующий SPARQL-запрос
- Абсолютно понятна логика работы каждой функции
- Простое добавление новых функций

## C. Ключевые технические принципы ver9d

### SPARQL-Driven Programming
- Бизнес-логика описывается декларативно через SPARQL-запросы
- JavaScript используется минимально — только для UI и координации
- Данные и их структура определяются онтологией
- Справочники и формы генерируются динамически на основе метаданных

### Архитектура хранилища
- **quadstore:** Хранение данных в формате квадов (subject, predicate, object, graph)
- **Графы:**
  - `vad:ptree` — дерево концептов процессов
  - `vad:rtree` — дерево концептов исполнителей
  - `vad:techtree` — технологическое приложение (методы, предикаты)
  - `vad:t_*` — схемы процессов (VADProcessDia)
  - `vad:vt_*` — виртуальные данные (автоматически вычисляемые)

### Модульная структура
- 12 функциональных модулей с чётким разделением ответственности
- Каждый модуль: UI (_ui.js), логика (_logic.js), SPARQL (_sparql.js)

## summary
- Семантический аналог для всего ARIS - подобного: http://www.bpm.processoffice.ru/  
- Тезис: от табличек и SQL к онтологиям (Protege) + tripleStore / quadstore + SPARQL + Reasoner и т.п. (Linked Data RDF\RDFS\OWL)  
- "Формальная семантика (LD) во всем" = semantic BPM/EA/CMDB/Web/Wiki и т.п.  
- Принцип Семантический ARIS (BPM): UI на js (и подобном), а логика на SPARQL (SPARQL-Driven подход)

## important
Версия ver9d сознательно имеет «перебор» со SPARQL окнами, чтобы:
-  продемонтсрировать реальную обработка на основе SPARQL-запросов (что "под капотом"), показать внутреннию "механику", в реальности SPARQL - запросы не будут визуализироваться и будут исполняться по кнопке
-  в целях откладки и тестирования MVP
